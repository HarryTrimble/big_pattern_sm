Transaction pattern
Before you start

Before starting a transaction, users need information to know if they can and should use it.

This pattern meet these needs with questions, calculated answers, to-do lists, showing rules.

1. How this pattern saves time and money

There are common issues and costs (failure demand) connected with not routing users to forms, activities and rules specific to the thing they want to do.

These issues and costs are:

  • Users cannot check all the things need in order to do something
  • Users are unclear what they need to do to not get in trouble with government for doing something
  • Users are unclear are to what rules apply to them
  • Users calling to ask if they need to fill in a form
  • Users calling to ask which forms they need to fill in
  • Users calling to ask what information they should put in a form
  • Processing applications users didn't need to fill in
  • Enforcement on filling out the wrong form
  • Enforcement on users misunderstanding the rules, then breaking them unknowingly

Teams have seen these cost of running their service drop, as a result of using this pattern.

2. Users needs this pattern meets

This pattern meets user needs. By using this pattern and meeting these user needs services can better meet policy intent and reduce running costs.

Before using a transaction users needs to know:

  • are they eligible to use a service e.g. carers allowance
  • do they have to use a service e.g drivers medical
  • what the full cost to use a service e.g firearm licence
  • the rules are for doing something e.g planning permission
  • what tasks are involved in applying or registering e.g. learn to drive

3. Features in a service this pattern can replace

Services with user need for routing questions, but not using the pattern have common features. By using the pattern, these features can be replaced.

These common features are:

  • "Pre app chats", where the service spends hours on the phone with users helping them prepare applications
  • Lots of guidance and instructions on how to fill in a form
  • Guidance and conditions split over multipple documents

4. How the pattern works

A service can route users using questions, data or both. This pattern uses questions.

By answering the questions a users can check all the things need in order to do something

The example comes from the Environment Agency. The service, Check if you need permission to do work on a river. They used the pattern to replace over a 100 pages of guidance split over 13 documents.

6. How to measure this pattern

Monitoring the performance of a pattern is essential: it allows the service teams using it or considering to, to know how well it works for their users and government.

The metrics measuring this pattern's performance are:

  • % of users unable to check all the things need in order to do something
  • Number of calls from users to ask what rules apply to them
  • Number of calls from users to ask if they need to fill in a form
  • Number of calls from users to ask which forms they need to fill in
  • Number of calls from users to ask what information they should put in a form
  • % of unnecessary applications a service processses
  • Amount of enforcement interactions on users who filled out the wrong form
  • Amount of enforcement interactions on users misunderstanding the rules, then breaking them unknowingly

To work out the total cost in time and money, times the measure above by:

  • cost per call
  • cost to process one application
  • cost per enforcement interacton

7. How to reuse this pattern

The pattern is not a technology.

The pattern is practical guidance to design routing questions for your service.

Involve the policy in the design of your routing questions.

If possible make the policy as the same time, in a multidisciplinary team of designers, policy makers, content designers and user research.
